{/* ─── Document Header ─────────────────────────────────────
    The mandatory first block of every Atto document. Declares
    document-level metadata (title, type, jurisdiction, effective
    date) and the cast of parties. Replaces the YAML-frontmatter
    concept used in earlier drafts: there is no `---` frontmatter
    in v0; everything that used to live there now lives inside
    ::atto-doc, which the parser treats as YAML between the open
    `::atto-doc` line and the closing `::`. */}

    template-literal expression triggers MDX whitespace
    normalization on continuation lines (see §06 alignedSource
    for the same workaround). */}

Every Atto document MUST begin with a `::atto-doc` block. It
declares document-level metadata and the cast of parties, and it
MUST appear before any body content, sections, or directives. The
content between `::atto-doc` and the closing `::` is parsed as
YAML.

<h3 id="document-header-parties">Parties schema</h3>

The `parties` list within `::atto-doc` is the **single source of
truth** for party identity in the document. The parser resolves
every party id reference in the document body — in
`::signature-block`, `required-signatory`, and `:::credit-check`
— against this declaration.

<FieldTable
  caption="parties entry fields"
  fields={[
    { name: 'id', type: 'string (kebab-case)', required: true, description: 'Machine-readable identifier. Referenced by ::signature-block, required-signatory, and chain entries. Lowercase letters, digits, and hyphens only.' },
    { name: 'name', type: 'string', required: true, description: 'Legal name of the party. Always the primary identifier in rendered output and chain entries.' },
    { name: 'role', type: 'string', required: true, description: 'Document-level role label. Displayed alongside the legal name in all rendered views. Cannot be overridden — only supplemented by a local label on a ::signature-block.' },
    { name: 'email', type: 'string', required: false, description: 'Contact address used by the platform to deliver recipient tokens and signing notifications.' },
  ]}

{/* ─── Parser rules ────────────────────────────────────── */}

<h3 id="document-header-rules">Parser rules</h3>

`::atto-doc` MUST appear as the first line of the file. The
parser MUST reject any document where:

- `::atto-doc` is absent,
- `::atto-doc` appears after body content,
- `::atto-doc` appears more than once,
- the `parties` list contains duplicate `id` values, or
- a `::signature-block`, `required-signatory`, or
  `:::credit-check` references an undeclared party id.

The content between `::atto-doc` and the closing `::` is parsed
as YAML. Keys are serialized alphabetically on output for
canonical byte stability. Unknown keys are preserved and
round-tripped without modification but produce no platform
behavior.

{/* ─── Reference & inheritance ─────────────────────────── */}

<h3 id="document-header-inheritance">Reference & inheritance</h3>

Downstream constructs reference parties by `id` and inherit the
canonical `role` label automatically. The party is declared once
in `::atto-doc.parties`; every other place it appears resolves
through that declaration.

<h4 style="font-family: var(--font-display); font-weight: 700; font-size: var(--text-base); margin-block: var(--space-5) var(--space-2);">Sections inherit the label</h4>

<pre class="code-block" set:html={sectionRefSource} />

The platform displays this section as **Required: Tenant,
Landlord** by resolving each id through `::atto-doc.parties`. The
`:::section` block does not re-declare the labels.

<h4 style="font-family: var(--font-display); font-weight: 700; font-size: var(--text-base); margin-block: var(--space-5) var(--space-2);">Signature blocks inherit the name</h4>

<pre class="code-block" set:html={signatureInheritSource} />

The rendered signature line reads **Tenant — Jordan Ellis**,
pulled from `::atto-doc.parties`. The signature block does not
re-state the name.

<h4 style="font-family: var(--font-display); font-weight: 700; font-size: var(--text-base); margin-block: var(--space-5) var(--space-2);">Local supplemental labels</h4>

The canonical `role` cannot be overridden. A `::signature-block`
MAY supplement it with a `label` for one specific signing
context — e.g., *Disclosing Party* in an NDA appendix. The render
appends the supplemental label after the role; it never replaces
it.

<AttoDemo layout="split" filename="supplemental-label.atto">
  <SignatureWidget
    slot="rendered"
    party="Tenant"
    signer="Jordan Ellis"

    order={1}

  <pre class="code-block" slot="source" set:html={signatureSupplementSource} />

The canonical role stays visible alongside the supplemental
label so a reader — or an auditor reviewing the chain — can
always trace the contextual label back to the underlying party
declared in `::atto-doc.parties`.
