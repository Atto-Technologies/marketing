A document that can be signed must be renderable identically at any
future time. The **Atto Font Registry Protocol** (AFRP) defines a
portable identifier grammar, a registry manifest format, and a
resolution algorithm that binds every executed document to the
exact font binaries present at signing.

This section is a normative summary of AFRP v1.0. The full
specification — including the JSON Schema, error code definitions,
governance procedures, and security analysis — lives at
`spec.atto.tech/afrp`.

{/* ─── Two tiers, three forms ─────────────────────────── */}

<h3 id="afrp-architecture">Two tiers, three forms</h3>

AFRP defines two namespace tiers operating under one resolution
protocol. The permissionless tier guarantees the governed tier
can never act as a gatekeeper for the system.

<FieldTable
  caption="AFRP tiers and identifier forms"
  fields={[
    { name: '@/Typeface', type: 'Form A · Tier 1', required: false, description: 'Canonical shorthand. No scope between @ and /. Resolves against the canonical Atto registry at https://fonts.atto.tech/. Portable across all compliant implementations without hardcoding a host.' },
    { name: '@scope/Typeface', type: 'Form B · Tier 1', required: false, description: 'Ratified scope shorthand. Tier 1 scopes are not self-serve — they are granted by invitation from the Atto Protocol Authority and ratified by the Technical Review Panel for foundational infrastructure providers.     { name: 'host/Typeface', type: 'Form C · Tier 2', required: false, description: 'Permissionless full URI. Any DNS host serving a compliant AFRP manifest is a valid registry. No registration, approval, or contact with any authority required. Equivalent openness to the Web itself.' },
  ]}
/>

Form A is distinguished from Form B by the absence of characters
between `@` and `/`. A resolver MUST NOT treat an unregistered
Form B scope as a fallback to Form A — it emits `SCOPE_NOT_FOUND`
and proceeds to the fallback chain.

#### Three paths to AFRP compliance

A font publisher has three ways to make their fonts available
inside AFRP-compliant documents. The architectural principle is
that **Tier 1 exists to protect namespace identity for permanent
infrastructure providers, not to gatekeep font publication**.

<ol class="spec-steps">
  <li>
    <strong>Self-host a Form C registry.</strong> Serve a compliant
    manifest at any DNS hostname you control. No registration,
    approval, or contact with any authority. Full technical
    responsibility rests with the publisher. See{' '}
    <a href="#afrp-manifest">Registry manifest</a> for the
    well-known endpoint and JSON Schema.
  </li>
  <li>
    <strong>Atto Hosted Registry Service (AHRS).</strong>{' '}
    Subscription-based managed hosting. The publisher provides
    font files and metadata; Atto operates the registry, computes
    integrity hashes, runs ARCTS, and serves a compliant manifest
    under <code>{'hosted.fonts.atto.tech/{slug}/'}</code> or a
    custom domain. Resolves as Form C. See{' '}
    <a href="#afrp-hosted">Hosted registry</a>.
  </li>
  <li>
    <strong>Tier 1 ratified scope.</strong> Reserved for
    foundational infrastructure providers meeting the permanence
    and identity criteria. Invitation-only ratification by the
    Technical Review Panel. Resolves as Form B{' '}
    <code>@scope/Typeface</code>. See{' '}
    <a href="#afrp-registration">Tier 1 ratification</a>.
  </li>
</ol>

The permissionless Form C tier guarantees that the Atto Protocol
Authority can never act as a gatekeeper for the system as a
whole — Tier 1 governs namespace identity only.

#### Sub-scope delegation

A registered scope MAY delegate sub-namespaces one level deep by
declaring a `delegations` map in its registry manifest. The parent
scope retains governance responsibility; the delegated registry
MUST pass ARCTS before the delegation is published.

Four-component identifiers like `@a/b/c/Name` are invalid per the
ABNF. Delegated registries cannot further sub-delegate.

unless the range itself is pre-release qualified. The version
portion follows SemVer 2.0.0 exactly.

#### Reserved names

<ul>
  <li>
    <strong>Scope reserved:</strong> <code>atto</code>,{' '}
    <code>official</code>, <code>authority</code>,{' '}
    <code>registry</code>, <code>root</code>, <code>sys</code>,{' '}
    <code>localhost</code>. Only the Atto Protocol Authority may
    use these.
  </li>
  <li>
    <strong>Version reserved:</strong> <code>latest</code> and{' '}

  </li>

</ul>

{/* ─── Axis qualifiers ────────────────────────────────── */}

<h3 id="afrp-axes">Axis qualifiers</h3>

Variable-font axes are selected by whitespace-separated qualifiers
of the form `tag=value` following the font identifier. Each
qualifier pins a single OpenType registered axis. Values absent
from a font reference default to the manifest's
`variable_axes.default` for that axis — or, if the manifest
declares none, to the OpenType-registered default (`wght=400`,
`opsz=12`, `ital=0`).

**Critical for execution integrity.** In EXECUTED state, *every*
axis value — including those not explicitly qualified at the
reference site — MUST be recorded in the execution metadata pin.
An absent qualifier means "use the manifest default," and that
default must be frozen at signing so later manifest edits cannot
alter visual output.

For non-variable fonts, only `wght` and `ital` are meaningful —
they drive static-style selection. All other axis qualifiers MUST
be ignored and an `AXIS_NOT_SUPPORTED` informational event is
emitted.

<h3 id="afrp-inline">Inline font references</h3>

notation <code>{'{font=...}'}</code>. An annotation accepts either
a declared id or a direct font reference (identifier + optional
axis qualifiers). Both forms resolve identically through the
algorithm in §08.7.

{/* ─── Registry manifest ──────────────────────────────── */}

<h3 id="afrp-manifest">Registry manifest</h3>

A compliant AFRP registry serves a registry-level manifest at the
well-known path `/.well-known/atto-font-registry.json`, returned
with `Content-Type: application/atto-font-manifest+json`.
Manifests MUST be served over HTTPS with valid TLS, carry
`Cache-Control` headers, and set
`Access-Control-Allow-Origin: *` for browser-based renderers.

Registries MAY additionally serve per-typeface manifests at
`/<TypefaceName>/manifest.json`. When available, resolvers SHOULD
prefer this endpoint to reduce bandwidth; a per-font response
takes precedence over the registry-level manifest for that
typeface.

<pre class="code-block" set:html={manifestSource} />

There is no manifest-level signature. Trust in the manifest flows
from TLS certificate validation, HSTS, and certificate transparency
monitoring — not from a JCS-canonicalized wire signature. Per-asset
`integrity` hashes guarantee binary authenticity downstream of the
manifest fetch.

<h3 id="afrp-resolution">Resolution algorithm</h3>

Resolution behavior depends on document state. An EXECUTED
document renders identically at any future time using exactly the
binaries present at signing.

  <thead>
    <tr>
      <th>Behavior</th>

      <th>EXECUTED mode</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><code>fallback_to_atto</code> default</td>
      <td><code>true</code></td>
      <td><code>false</code></td>
    </tr>
    <tr>
      <td>System font substitution</td>
      <td>Permitted</td>

    <tr>

      <td>MUST be applied</td>
    </tr>

The 11-step algorithm in AFRP §5.2 is non-negotiable. Every
compliant resolver MUST:

<ol class="spec-steps">
  <li>Parse the font reference per the §3.2 ABNF. Fail fast on syntax errors.</li>
  <li>Translate reserved aliases (<code>weight=</code> → <code>wght=</code>, <code>style=italic</code> → <code>ital=1</code>).</li>
  <li>Classify as Form A / Form B / Form C and derive the registry base URL.</li>
  <li>Fetch the manifest (see three-step discovery below), validate schema, protocol version, integrity-field format, and URL scheme.</li>
  <li>Locate the font entry by typeface name.</li>
  <li>Select a version satisfying the specifier per SemVer precedence.</li>
  <li>Select a style entry, then a format entry (<code>woff2</code> preferred).</li>
  <li>Download the binary, compute SHA-256 over the decompressed body, compare constant-time against <code>integrity</code>.</li>
  <li><strong>EXECUTED only:</strong> compare against <code>full_integrity</code> from the pin. Mismatch → <code>INTEGRITY_VIOLATION</code>, halt rendering.</li>
  <li>If <code>license_info.commerce</code> is present, validate the license token or acquire a <code>use_receipt</code> from <code>payment_endpoint</code>.</li>
  <li>Cache the verified asset keyed by <code>(registry, typeface, version, format, weight, style, integrity)</code>.</li>
</ol>

#### Form C manifest discovery

For Form C identifiers, the resolver attempts manifest discovery
in three ordered steps. The first success ends the search:

<ol class="spec-steps">
  <li>
    <strong>Well-known:</strong> GET{' '}
    <code>https://host/.well-known/atto-font-registry.json</code>.
    If the typeface appears in the <code>fonts[]</code> array,
    resolution proceeds against this manifest.
  </li>
  <li>
    <strong>Per-typeface:</strong> GET{' '}
    <code>https://host/&lt;TypefaceName&gt;/manifest.json</code>.
    Takes precedence over the registry-level manifest when
    available.
  </li>
  <li>
    <strong>Inline:</strong> use the{' '}
    <code>inline_manifest</code> declared in the document's{' '}
    <code>fonts[]</code> block. Final fallback. MUST match
    execution metadata in EXECUTED documents.
  </li>
</ol>

All three attempts failing → `REGISTRY_UNREACHABLE`. Resolvers
MUST NOT construct heuristic URLs based on scope names. TLS
relaxations apply only to `localhost` and loopback addresses in
DRAFT mode.

{/* ─── Integrity & pin ────────────────────────────────── */}

<h3 id="afrp-integrity">Integrity &amp; execution pin</h3>

AFRP uses **SHA-256** subresource integrity hashes throughout, in
the canonical SRI format `sha256-<base64>`. Hash comparison MUST
use constant-time equality to prevent timing side-channels. A
resolver MUST NOT cache or render with any asset whose hash fails
verification.

When an AD is sealed to EXECUTED state, the signer freezes a
complete execution pin for each declared font. The pin is covered
by the document signature — tampering invalidates the whole
document.

<FieldTable
  caption="Execution pin fields"
  fields={[

**Two-layer integrity.** `subset_integrity` alone is insufficient
— it proves only that the embedded binary is unaltered, not that
it was derived from the legitimate source. Both `full_integrity`
and `subset_integrity` MUST be verified together for a complete
integrity assertion.

{/* ─── Fallback policy ───────────────────────────────── */}

<h3 id="afrp-fallback">Fallback policy</h3>

When primary resolution fails with an ERROR-level code, the
resolver evaluates a four-step fallback chain. The first success
ends the chain:

<ol class="spec-steps">
  <li>
    <strong>F1 — Document-declared fallback.</strong> Each entry
    in the declaration's <code>fallback[]</code> array is resolved
    sequentially through the full §5.2 algorithm.
  </li>
  <li>
    <strong>F2 — Implicit Atto fallback (Form B only).</strong>{' '}
    If the primary is Form B and <code>fallback_to_atto</code> is
    true, attempt to resolve <code>{'@/{TypefaceName}'}</code>{' '}
    against the canonical Atto registry. Form A and Form C MUST
    NOT trigger this step.
  </li>
  <li>
    <strong>F3 — System font reference.</strong> If{' '}
    <code>fallback[]</code> contains a <code>system:</code> entry
    (§7.3) and prior attempts failed, substitute the system font.{' '}
    <em>DRAFT only.</em>
  </li>
  <li>
    <strong>F4 — Terminal failure.</strong> DRAFT: system
    sans-serif + <code>FONT_UNRESOLVED</code> warning. EXECUTED:
    visible error indicator, rendering halted for that region.
    MUST NOT substitute any font.
  </li>
</ol>

#### System font references

System fonts use the `system:` prefix and are valid only as
fallback entries. They MUST NOT appear as primary identifiers and
MUST NOT be the sole font entry in a document intended for
EXECUTED state — system fonts vary across platforms and cannot be
integrity-pinned.

  caption="System font references"
  fields={[
    { name: 'system:sans-serif', type: 'fallback-only', required: false, description: 'System default proportional sans-serif. Equivalent to CSS font-family: sans-serif.' },
    { name: 'system:serif', type: 'fallback-only', required: false, description: 'System default proportional serif.' },
    { name: 'system:monospace', type: 'fallback-only', required: false, description: 'System default monospace.' },
    { name: 'system:ui', type: 'fallback-only', required: false, description: 'System default UI font (San Francisco, Segoe UI, etc.).' },

/>

**Offline fallback.** Every compliant AFRP renderer MUST bundle a
copy of `@/Lineal` with an integrity hash baked into the renderer
build. DRAFT-mode offline resolution falls back first to this
bundled binary (emitting `FONT_SUBSTITUTED`), then to the host
system's sans-serif. EXECUTED mode forbids all offline
substitution.

<h3 id="afrp-commerce">Font Commerce Layer</h3>

AFRP embeds a native payments surface for fonts. A
`license_info.commerce` block in the manifest declares one of
five pricing models. Resolvers enforce the commerce contract
during resolution (§5.2 step 10); the resulting `use_receipt` is
embedded in the executed document's pin.

<h3 id="afrp-registration">Tier 1 ratification</h3>

**Tier 1 `@scope` grants are not available by self-serve
application.** They are initiated by the Atto Protocol Authority
or by invitation, reviewed against the criteria below, and
ratified by the Technical Review Panel. An entity may express
interest by emailing
[`registry@atto.tech`](mailto:registry@atto.tech), but interest
does not constitute an application and does not guarantee review.
Tier 1 is reserved for foundational infrastructure providers —
for self-serve hosting, see the
[Hosted Registry Service](#afrp-hosted).

The criteria govern identity, permanence, transparency, and
technical compliance — none of them are content-restrictive.
Catalog depth and originality were criteria in earlier drafts;
they have been removed because they are content determinations
the Authority is not positioned to make equitably across all
cultural and organizational contexts. A Tier 1 grant requires
satisfaction of all five criteria below.

<ol class="spec-steps">
  <li><strong>Infrastructure permanence.</strong> A foundational font infrastructure provider whose registry is expected to remain operational, publicly accessible, and independently maintained indefinitely. Evidence: organizational age of at least five years, institutional backing or endowment, a public commitment to long-term stewardship.</li>
  <li><strong>Identity verification.</strong> The scope name MUST correspond to the applicant's verified legal or organizational identity (e.g. <code>@adobe</code> for Adobe Inc., <code>@velvetyne</code> for Velvetyne Type Foundry). Verified by trademark, WHOIS-verified domain, or equivalent documentary evidence. Invented or speculative scope names are not eligible.</li>
  <li><strong>License transparency.</strong> Every font served MUST include machine-readable SPDX metadata. Proprietary licenses MUST accurately declare <code>commercial_use</code>, <code>embedding_allowed</code>, and <code>web_embedding_allowed</code>. False declarations are grounds for revocation.</li>
  <li><strong>Protocol compliance.</strong> Registry MUST pass the <strong>ARCTS</strong> test suite — schema conformance, per-asset integrity verification, TLS validity, version resolution behavior, cache headers, CORS. A passing report dated within 30 days of submission is required.</li>
  <li><strong>Code of Conduct.</strong> Digitally signed agreement. Prohibits malware payloads, obfuscated binaries, homoglyph phishing fonts, false metadata, and using a granted scope inconsistently with the applicant's verified identity.</li>
</ol>

#### Fee structure

Tier 1 grants are not tiered by catalog size or commercial
status. The fee reflects the ongoing operational cost of
maintaining the scope holder's entry in the shorthand registry,
running monthly ARCTS checks, weekly integrity spot-checks, and
quarterly license audits — the same cost regardless of whether
the holder is a two-person foundry or a global platform.

#### Ratification procedure

Express interest at

the three-person Technical Review Panel with balanced open-source
and commercial representation; panel decisions are final.
